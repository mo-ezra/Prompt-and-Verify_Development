PaV

Prompt-and-Verify Development

An AI-assisted software method where engineers use natural-language prompts to generate code and always pass it through human review, automated tests, and standards before acceptance.

Canonical home: GitHub repository

Definition

Prompt-and-Verify Development (PaV) is a disciplined workflow for AI-assisted coding: the AI drafts; the engineer verifies with code review, tests, linters, and security checks. If verification fails, the engineer revises prompts or hand-edits until it passes. Nothing merges without verification.

How PaV differs from “vibe coding”

AspectVibe codingPaV
Review depthMinimal / opportunisticMandatory, deep
Quality gatesMay skip or deferTests, lint, security, CI required
Risk posturePrototype-firstReliability-first
Suitable forFast prototypingProduction & regulated teams

Core principles

  1. Single source of truth: specs & tests define correctness.
  2. Human accountability: engineers own merges and outcomes.
  3. Tooling neutrality: PaV works with any LLM or IDE.
  4. Defense in depth: static analysis, SCA, secrets scan, SAST/DAST where relevant.
  5. Repeatable prompts: prompts live with code (docs or repo).

Reference workflow

  1. Frame: write/attach a brief spec or tests first.
  2. Prompt: ask the LLM for code changes (scoped, testable).
  3. Build locally: run unit tests, lints, type checks.
  4. Review: human code review → request fixes or edit.
  5. Gate: CI must pass verification checks before merge.
  6. Document: commit prompt & decisions (ADR/changelog).

Prompt examples

Implement a pure function `normalizePhone(input: string): string`:
- E.164 format; keep leading '+'; strip spaces/dashes; validate country code.
- Add tests for US/UK/NG valid and invalid inputs.
- Do not add dependencies. Keep under 60 lines.
- Existing stack: TypeScript + Vitest. Place tests in tests/normalizePhone.test.ts.
Refactor `UserRepo.save()` to be idempotent:
- If a row exists with same email, update instead of insert.
- Add migration script and integration tests.
- Postgres 15, Node.js Prisma ORM.
- Ensure transactions and rollback on failure.

Implementation checklist

  • Repo contains CONTRIBUTING.md describing PaV gates.
  • CI enforces: tests, lints, type-check, coverage threshold, security scans.
  • Prompts stored under docs/prompts/ or in PR description.
  • ADR for notable AI-assisted design decisions.

Minimal GitHub Actions (Node example)

Copy and adapt to your stack.

# .github/workflows/pav.yml
name: PaV Gates
on:
  pull_request:
    branches: [ main ]
jobs:
  verify:
    runs-on: ubuntu-latest
    steps:
      - uses: actions/checkout@v4
      - uses: actions/setup-node@v4
        with: { node-version: '20' }
      - run: npm ci
      - run: npm run lint
      - run: npm run typecheck --if-present
      - run: npm test -- --ci --reporters=default --coverage
      - name: Block if coverage & audits fail
        run: |
          npm run coverage:check || exit 1
          npm audit --audit-level=high || true
      - name: Secret scan
        uses: trufflesecurity/trufflehog@v3
        with:
          path: .
          base: ${{ github.event.pull_request.base.sha }}
          head: ${{ github.event.pull_request.head.sha }}

FAQ

What problems does PaV solve?

It enables speed from LLMs without sacrificing reliability, security, or maintainability—by making verification non-negotiable.

Is PaV tool- or vendor-specific?

No. It’s a process. Use any LLM, editor, or CI as long as verification gates are enforced.

Can PaV work in regulated environments?

Yes—PaV maps well to existing controls (peer review, test evidence, SoD). Keep prompt history and decisions in the repo for auditability.

How do I start?

  1. Adopt the CI gates above.
  2. Add a CONTRIBUTING.md with PaV rules.
  3. Store prompts with the code or PR.
  4. Pilot on a small service; iterate.

Is PaV only for “fixes” or also for net-new code?

Both. The key is always the same: code is accepted only after verification passes.
